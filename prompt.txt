> 此文件可直接整份複製貼給具備程式開發能力的 AI Agent（例如 Claude Code、Cursor 等），讓其依此規格獨立完成整套系統的設計與開發。

## 一、角色與任務

你是一位資深全端工程師暨產品設計師。請依照以下完整需求規格，獨立設計並開發一套「智慧工廠監控平台（Smart Factory Monitoring Platform）」的前端 Demo / POC 原型系統，用於對主管或客戶展示工廠數位化管理的完整概念。

**核心原則：**
- 這是 **Demo / POC**，以**前端為主**，所有資料皆使用**模擬假資料**，不需串接真實後端、資料庫或硬體設備
- 但畫面、互動與資料邏輯需要做到**以假當真**的程度 —— 數字要合理、趨勢要像真實工廠、操作要順暢，讓人一看就覺得「這是一套正在運作的系統」
- 介面語言為**純繁體中文**（所有選單、按鈕、表單標籤、提示訊息、圖表座標軸與單位皆需中文化，慣用工廠/製造業術語）
- 技術棧**不指定**，由你依需求自行選擇最合適的方案（前端框架、UI 元件庫、圖表庫等）。**動工前請先用一段話說明你的技術選型與理由**，再開始實作
- 最終交付物必須是**可直接執行、可實際互動操作**的網頁應用程式，不是靜態截圖或純設計稿

## 三、整體 UI/UX 設計原則

- **視覺調性**：工業控制室／SCADA 監控風格，專業、冷靜、資訊密度高但不雜亂。建議深色主題為主（或提供深/淺色切換），數據用高對比色突顯。
- **版面結構**：左側可收合導覽選單 + 頂部 Header（廠區/產線選擇器、日期區間選擇、通知鈴、使用者資訊）+ 中央內容區。
- **狀態色彩語意需全站一致**：綠＝正常／達標、黃＝警告／需關注、紅＝異常／停機、灰＝閒置／離線。所有狀態標籤（Badge）、圖表配色都應遵循此規則。
- **圖表多樣化**：依資料性質選用折線圖（趨勢）、長條圖（比較）、圓餅/甜甜圈圖（佔比）、Gauge 儀表（即時達成率）、Pareto 圖（不良/異常排序）、甘特圖或時間軸（工單、保養排程）、看板（Kanban，問題追蹤、改善流程）、日曆（保養排程）。
- **即時感模擬**：「即時產線」「設備狀態監控」等頁面需用計時器（如每 3-5 秒）局部更新數值，營造真實監控畫面的動態感，而非完全靜態。
- **RWD**：以桌面大螢幕（監控牆情境）為主要最佳化對象，同時兼顧一般筆電瀏覽器視窗大小。
- **共用元件**：KPI 卡片、狀態 Badge、資料表格（含排序/篩選/分頁）、圖表容器、Modal/Drawer 表單、看板卡片等應抽成可重用元件。

## 四、各模組功能規格

### 1. 生產監控

#### 1.1 即時產線
- 以卡片或看板形式呈現所有產線，每條產線顯示：產線名稱、目前生產中產品、目前狀態（運轉中／閒置／停機／換線中）、目前班次與操作員、實際產量 vs 目標產量（含達成率進度條）、Takt Time 與實際週期時間比較。
- 可選配簡化版產線站點示意圖（如一排站點方塊，標示各站點狀態）。
- 點擊產線可展開/進入詳細頁，顯示近 1 小時產量趨勢與停機事件列表。
- 數值需以計時器定時模擬更新，呈現「即時」感。

#### 1.2 工單管理
- 工單列表：工單號、產品名稱、計畫數量、已完成數量、預計開始/結束時間、優先序、狀態（待排程／生產中／暫停／已完成）。
- 提供依產線分組的甘特圖／時間軸視圖，呈現工單排程與重疊狀況。
- 支援新增、編輯、篩選（依產線/狀態/日期）、搜尋工單。
- 工單詳細頁：進度條、實際 vs 計畫產出趨勢圖、相關異常或品質事件關聯。

#### 1.3 產量分析
- KPI 卡片：今日總產量、達成率、本週/本月累計產量、與去期比較。
- 趨勢圖：依日/週/月切換的產量趨勢（可疊加目標線）。
- 依產線、依產品的產量比較長條圖。
- 簡易 Pareto 分析：產量佔比前幾大產品/產線。

### 2. 品質管理

#### 2.1 良率
- 即時良率 Gauge（整廠或選定產線），並標示目標門檻線。
- 良率趨勢圖：可依產線、產品、時間區間切換比較。
- 良率排行（依產線/產品排序），快速找出表現落後者。

#### 2.2 不良分析
- 不良類型 Pareto 圖（依發生次數或數量排序，標示累積百分比線）。
- 不良趨勢圖：依時間呈現各不良類型發生趨勢。
- 不良明細表：時間、產線、站點、班次、不良類型、數量、判定人員（可預留照片縮圖佔位）。
- 可依不良類型/產線/時間篩選交叉分析。

#### 2.3 CAPA（矯正與預防措施）
- CAPA 案件列表：案件編號、來源（關聯不良/異常事件）、嚴重度、負責人、狀態（開立／調查中／審核中／已關閉）、預計關閉日期。
- CAPA 詳細表單，包含問題描述、根因分析、暫時對策（containment）、矯正措施、預防措施、效果驗證等步驟區塊（類似 8D 流程）。
- CAPA 效果追蹤儀表：已關閉案件的後續不良率變化比較圖。

### 3. 設備管理

#### 3.1 狀態監控
- 設備清單／網格視圖：設備名稱、所屬產線、目前狀態（運轉／閒置／故障／保養中）、運轉時數。
- 設備詳細頁：模擬感測數據（如溫度、振動、轉速、電流等）以小型即時折線圖或數值卡呈現，並模擬異常告警（超出門檻時轉紅並產生告警記錄）。
- 告警歷史列表：時間、設備、告警內容、處理狀態。

#### 3.2 OEE（整體設備效率）
- OEE 公式拆解卡片：可用率（Availability）× 效能（Performance）× 良率（Quality）= OEE，並顯示各分項數值與整體 OEE。
- OEE 趨勢圖：依日/週呈現整廠或選定設備的 OEE 變化。
- 設備 OEE 排行榜，快速找出表現最差的設備。
- 損失原因分析（圓餅或長條圖）：依停機原因（換線、故障、缺料、其他）拆解可用率損失。

#### 3.3 預防保養
- 保養排程日曆視圖：標示各設備下次保養日期。
- 保養計畫列表：設備、保養週期、上次保養日期、下次到期日期、負責人，並以顏色標示即將到期/已逾期。
- 保養工單追蹤：狀態（已排程／執行中／已完成），可展開查看保養項目檢查表（Checklist）。

### 4. 追溯系統

#### 4.1 批次
- 批次列表：批次號、產品、生產日期、產線、數量、目前狀態（生產中／已完成／已出貨／已封存）。
- 支援依批次號、產品、日期區間搜尋與篩選。
- 點擊批次可進入該批次的生產履歷詳細頁。

#### 4.2 生產履歷
- 以時間軸或流程圖形式呈現單一批次的完整履歷：使用的原物料批號、生產產線與設備、操作人員、各站點作業時間、品質檢驗結果（良率/不良記錄）。
- 提供簡易的「正向追溯」（原料→成品）與「反向追溯」（成品→原料）視覺化呈現（可用樹狀圖或流程圖示意）。
- 提供匯出/列印追溯報告的按鈕（demo 中可僅為觸發匯出動作，不需真實檔案產出邏輯，或產出簡單 PDF/CSV 皆可）。

### 5. 異常管理

#### 5.1 問題追蹤
- 異常事件列表與看板（Kanban）雙視圖切換：欄位包含案件編號、類型（設備/品質/物料/安全等）、嚴重度、關聯產線/設備、回報人、狀態（新建／已指派／處理中／已解決／已關閉）。
- 看板視圖以狀態分欄，可拖曳卡片切換狀態（demo 可用簡單拖放或下拉切換模擬）。
- 案件詳細頁：時間軸記錄（回報→指派→處理→解決）、留言/備註區、SLA 倒數計時提示（如距離承諾處理時限剩餘時間）。

#### 5.2 改善流程
- 改善專案列表：依 PDCA（計畫／執行／檢核／行動）階段呈現專案進度。
- 看板或進度條呈現各專案目前所處階段。
- 改善前後效果對比卡片（如改善前後的不良率、停機時間等指標對比），可與品質管理/設備管理模組資料呼應。

### 6. Dashboard（總覽儀表板）
- 整合全廠關鍵指標的 KPI 卡片：整體 OEE、整體良率、今日產量達成率、進行中異常案件數、待處理 CAPA 數量等。
- 各產線/設備狀態總覽（如燈號矩陣或地圖式佈局）。
- 多個迷你趨勢圖（Sparkline）呈現近期關鍵指標走勢。
- 告警/通知摘要面板，列出最新的異常與待辦事項，點擊可導向對應模組詳細頁。

## 五、模擬資料規範

- 建立**集中管理的 mock data 模組**（例如獨立的 data 檔案或 mock service 層），方便未來替換為真實 API 而不需大幅修改頁面邏輯。
- 基本資料規模建議：至少 4-6 條產線、8-12 種產品、15-25 台設備、近 30 天的歷史資料量，足以展示篩選/趨勢/排行等功能，但不需過度龐大。
- 數值需符合製造業常見區間，增加真實感，例如：
  - OEE：常見範圍約 50%-85%，世界級標竿約 85% 以上
  - 良率：一般約 92%-99%
  - 設備可用率：約 70%-95%
  - 產量達成率：約 85%-105%（可有少數超標或落後案例，避免所有數字都太完美）
- 對於「即時」類頁面，數值應有小幅隨機波動（而非每次重整就大幅跳動），模擬真實感測資料的微幅起伏。
- 異常/不良事件的發生頻率分布應符合 80/20 法則（少數類型佔大多數案例），讓 Pareto 圖呈現有意義的形狀。

## 六、非功能需求

- 程式碼模組化、共用元件可重用，檔案/資料夾結構清晰，命名具一致性。
- 各頁面需有基本的 loading 與空狀態（empty state）呈現，提升完整度。
- 頁面切換與互動需流暢，避免不必要的重渲染或卡頓。
- 圖表與表格在資料量較大時需有分頁、虛擬滾動或篩選機制，避免畫面卡死。

## 七、建議開發步驟

1. 確認技術棧並初始化專案，建立基礎路由結構。
2. 建立整體版面骨架（導覽列、Header、共用設計系統：色彩、字型、間距、共用元件）。
3. 建立集中式 mock data 模組與資料產生邏輯。
4. 依序實作六大模組頁面，建議順序：Dashboard → 生產監控 → 設備管理 → 品質管理 → 追溯系統 → 異常管理（先做總覽建立資料骨架，再逐步補齊細節模組）。
5. 為「即時產線」「設備狀態監控」等頁面加入定時更新模擬效果。
6. 全站視覺收尾、互動細節打磨（hover、轉場、響應式調整）。
7. 自我檢查：依本文件逐一核對各模組功能是否完整呈現，並模擬一次完整 Demo 操作流程。
